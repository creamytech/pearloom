## Executive Summary

Pearloom's foundation is real: 101 routes, 369 components, 36 migrations, a clear three-pillar product (Compose / Conduct / Remember), 28 event types catalogued, and a brand system documented to a level most production companies never reach. The core happy paths work — wizard generates sites, editor saves, sites publish, guests RSVP, Pear assists. **But the gap between "works in dev" and "I trust this with a paying customer's wedding day" is wider than the commit log suggests.**

The biggest immediate problem is **verification**: tests fail to run at all, and the few that exist cover utilities rather than the money flows. The biggest **security** problem is a real one: a Stripe webhook race condition + a missing ownership check on `/api/invite` GET/DELETE. The biggest **legal** problem is that the privacy page promises GDPR functionality that isn't implemented. The recent look-and-feel sprint has been productive but is decorating a house with a leaky roof.

**Headline call:** a focused **8-week production hardening push**:
- **Week 1** stops the bleeding (fix tests, plug the three security holes, ship GDPR endpoints)
- **Weeks 2–4** build the regression net around the five money flows
- **Weeks 5–6** do production-grade infra (Sentry, Redis rate limiting, env hygiene)
- **Weeks 7–8** close the polish + chrome gaps already in flight

The 23 beta-event-types and real-time broadcast work in CLAUDE-PRODUCT.md §7 should wait until that hardening lands, or they'll add surface area to an unverified system.

## Top 5 Risks (in priority order)

1. ~~**Stripe webhook is not idempotent under retries.**~~ **FIXED 2026-05-29.** Migration [20260529_registry_claims_idempotency.sql](supabase/migrations/20260529_registry_claims_idempotency.sql) adds `stripe_session_id` to `registry_item_claims` with a partial unique index; the actual webhook ([src/app/api/stripe/webhook/route.ts](src/app/api/stripe/webhook/route.ts), not the `billing/webhook` path the audit pointed at) now does a pre-check + catches Postgres 23505 unique-violation as benign, and links the claim to the payment row via `payment_id`. Bonus: the `quantity_claimed` bump now runs exactly once per Stripe session, fixing a second race the audit didn't flag.

2. **Test suite is non-functional and zero e2e coverage on the money flows.** 14 vitest files fail with a config error. No e2e exists for wizard, publish, RSVP, payment, co-host, or day-of broadcast. The Studio sprint added e2e tests but they mock the very routes that need exercise. Every commit currently ships on faith.

3. ~~**Privacy page promises GDPR export and account deletion that isn't implemented.**~~ **FIXED 2026-05-29 (Phase 1.5).** Shipped [POST /api/user/export-data](src/app/api/user/export-data/route.ts) (10/hr rate limit, returns downloadable JSON of every owned row across ~25 tables) + [POST /api/user/delete-account](src/app/api/user/delete-account/route.ts) (3/24h rate limit, requires email-echo confirmation, hard-deletes all sites+children then user-keyed rows). Wired into the existing [DashSettings](src/components/marketing/design/dash/DashSettings.tsx) "Weave a backup" and "Delicate actions" sections (which were UI stubs pointing at a non-existent endpoint + a dead button). Privacy page now deep-links via `?section=export` and `?section=danger`.

4. ~~**Ownership check missing on `/api/invite` GET and DELETE.**~~ **FIXED 2026-05-29.** Added `ownsSite()` helper + gates on POST, GET, DELETE in [src/app/api/invite/route.ts](src/app/api/invite/route.ts). POST was also vulnerable (audit only flagged GET/DELETE) — any authed user could mint invites for any siteId. All three methods now 403 non-owners.

5. **In-memory rate limiting won't survive multi-instance deployment on Vercel.** Every AI endpoint (`/api/generate`, `/api/photos/stylize`, `/api/inline-rewrite`) shares the same in-memory limiter. First scale event lets bad actors burn the Gemini + Claude budgets. → [src/lib/rate-limit.ts](src/lib/rate-limit.ts) — *partial fix: plan-tier gates now in place (Phase 3.3) so legitimate over-cap users can't burn the budget; Redis swap is still required for multi-region survival.*

## Footnote: methodology

Each lens agent read CLAUDE-DESIGN.md / CLAUDE-PRODUCT.md / BRAND.md first, then produced 8-18 findings with file-path + line-range evidence. Every finding got **two adversarial verifiers** in parallel:
- A **code-reality** verifier (Explore tools) read the cited file at the cited lines and checked the described state was still true and not already fixed in recent commits.
- A **user-impact** verifier asked "does this gap actually hurt a real host or guest, or is it engineering hygiene?" — and could downgrade severity.

A finding survived only if **both** verifiers returned `confirmed` or `partial`. Severity was conservatively revised to the lower of the two — so what's in this doc is what survived skepticism, not what an over-eager auditor wanted to flag.

Workflow run: `wf_349bd46c-30b` · 286 agents · 26 minutes · 20.2M tokens.
